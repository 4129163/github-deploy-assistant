# 集成测试与端到端测试架构设计

## 概述
本测试架构旨在为GitHub Deploy Assistant项目提供完整的集成测试和端到端测试能力，验证"克隆→安装→启动"全流程，并确保测试覆盖率≥80%。

## 测试类型

### 1. 集成测试 (`test/integration/`)
- **目标**: 验证模块间的集成和业务流程
- **范围**: 服务层、API层、数据库交互
- **工具**: Jest + Supertest

### 2. 端到端测试 (`test/e2e/`)
- **目标**: 验证完整的用户工作流
- **范围**: 从用户界面到后端服务的完整流程
- **工具**: Playwright

### 3. 临时目录测试 (`test/utils/temp-project/`)
- **目标**: 在临时目录中启动真实Node.js项目
- **范围**: 验证项目部署全流程
- **工具**: Node.js child_process + fs-extra

## 目录结构

```
test/
├── integration/           # 集成测试
│   ├── api/              # API集成测试
│   ├── services/         # 服务层集成测试
│   ├── utils/            # 工具函数集成测试
│   └── setup.js          # 集成测试设置
├── e2e/                  # 端到端测试
│   ├── workflows/        # 工作流测试
│   ├── ui/              # 用户界面测试
│   ├── playwright.config.js
│   └── setup.js         # Playwright设置
├── utils/               # 测试工具
│   ├── temp-project/    # 临时项目模板
│   ├── test-helpers.js  # 测试辅助函数
│   └── test-server.js   # 测试服务器
└── coverage/            # 覆盖率报告
```

## 测试流程设计

### 集成测试流程
1. **环境准备**: 启动测试服务器，初始化测试数据库
2. **API测试**: 验证REST API端点
3. **服务测试**: 验证业务逻辑
4. **清理**: 清理测试数据，关闭服务器

### 端到端测试流程
1. **浏览器启动**: 启动Playwright浏览器
2. **UI导航**: 模拟用户界面操作
3. **流程验证**: 验证"克隆→安装→启动"全流程
4. **断言**: 验证预期结果

### 临时目录测试流程
1. **创建临时目录**: 使用fs-extra创建临时项目目录
2. **初始化项目**: 创建package.json和基本文件
3. **执行部署**: 调用部署助手API
4. **验证结果**: 检查项目是否成功启动

## 覆盖率目标

- **行覆盖率**: ≥80%
- **分支覆盖率**: ≥70%
- **函数覆盖率**: ≥85%
- **语句覆盖率**: ≥80%

## 影响模块

- `test/` 目录: 新增集成测试和端到端测试
- `src/__tests__/`: 补充单元测试
- `package.json`: 新增测试脚本
- `playwright.config.js`: 新增Playwright配置
- `jest.config.js`: 更新测试配置

## 技术栈

- **测试框架**: Jest
- **HTTP测试**: Supertest
- **端到端测试**: Playwright
- **覆盖率**: Jest Coverage
- **临时文件**: fs-extra, tmp-promise
- **进程管理**: child_process
